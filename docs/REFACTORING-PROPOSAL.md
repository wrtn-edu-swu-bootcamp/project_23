# 프로젝트 리팩토링 제안

현재 NAVA 프로젝트는 Next.js App Router를 사용하고 있으며, 각 탭이 별도의 라우트로 구성되어 있습니다:
- `/folio` - FOLIO 탭
- `/note` - NOTE 탭
- `/planner` - PLANNER 탭
- `/article` - ARTICLE 탭
- `/vv` - VV 탭

## 현재 구조의 장점

1. **SEO 최적화**: 각 페이지가 고유한 URL을 가짐
2. **브라우저 히스토리**: 뒤로가기/앞으로가기 지원
3. **코드 분할**: 각 페이지가 필요할 때만 로드됨
4. **URL 공유**: 특정 탭을 직접 공유 가능

## 제안하는 리팩토링 방향

### 옵션 1: 현재 구조 유지 + 공통 컴포넌트 분리 (권장)
- 각 탭의 라우트 구조 유지
- 공통 로직을 커스텀 훅으로 분리
- 재사용 가능한 UI 컴포넌트 생성

**장점:**
- Next.js 베스트 프랙티스 준수
- 성능 최적화
- 유지보수 용이

### 옵션 2: 단일 페이지 애플리케이션 (SPA) 구조
- 모든 탭을 메인 페이지에서 state로 관리
- `components/tabs/` 디렉토리에 탭 컴포넌트 분리
- URL은 변경되지 않음

**단점:**
- SEO 불리
- 모든 탭이 한 번에 로드됨 (성능 저하)
- 브라우저 히스토리 작동 안함
- URL 공유 불가

## 추천 방법: 옵션 1

공통 로직과 컴포넌트를 분리하여 코드 재사용성을 높이고, 
현재의 라우팅 구조는 그대로 유지하는 것을 권장합니다.

다음을 생성하겠습니다:
1. `hooks/` - 공통 로직 (localStorage, favorites 등)
2. `components/common/` - 재사용 가능한 UI 컴포넌트
3. `lib/types.ts` - TypeScript 타입 정의

이 방법이 더 좋을까요, 아니면 여전히 옵션 2(SPA 구조)를 원하시나요?
